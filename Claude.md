Development Rules

1. Analysis Before Implementation
    Do not guess. If anything is unclear, ask clarifying questions.
    State assumptions explicitly and propose simple solutions before writing code.
2. Minimalism
    Write code only for the current task. No future-proofing unless explicitly requested.
    If you see a way to simplify or reduce the code, do it.
3. Targeted Changes
    Modify only the code directly related to the task. Do not touch adjacent functions or any other logic unless explicitly requested.
    Follow the project's established coding style.
    Clean up after yourself: if functionality is no longer used, remove it. Remove only those dependencies that became unnecessary as a direct result of your changes.
4. Complex Tasks — Use Composition
    For complex tasks, provide a plan that includes:
    The actions to be taken.
    The verification method that will be used to validate the result.
5. Working process
    Write documentation after each phase or update existing one if it nessesary
    Write test case which cover real scenarion not fiction
